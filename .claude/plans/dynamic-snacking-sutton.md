# Plan: k8s Folder Rewrite — No Hardcoding, Template Reuse, Secrets via k8s-manager

## Context

Three problems exist in the current k8s charts:
1. **Hardcoded credentials** — passwords (Jenkins, MongoDB, Keycloak, Grafana, SonarQube token, Kong RSA key) are committed in `values.yaml` files, making the repo a credential store.
2. **No single source of truth for secrets** — `k8s-manager/env/credentials.env` holds credentials, but `values.yaml` still has duplicate hardcoded defaults; `setup.go` passes only some via `--set` (brittle with `@` in passwords).
3. **Copy-paste templates** — vault-agent-config and vault-agent sidecar containers are duplicated across backend/frontend/learning with 90%+ identical content.

**Goal:** All credentials live only in `credentials.yaml` (YAML, not .env) → k8s-manager generates a `values-secrets.yaml` at deploy time → Helm picks it up. Charts have empty defaults for all sensitive fields. Shared Helm named templates eliminate duplication.

## Phase 2 — k8s-manager: Migrate to credentials.yaml + Generate values-secrets.yaml

### `k8s-manager/env/credentials.yaml` (replaces credentials.env)
New YAML format — supports multiline values natively (critical for Kong RSA PEM key):
```yaml
jenkins:
  adminPassword: ""
  githubUser: ""
  githubPat: ""       # must set
  nexusUser: admin
  nexusPat: ""        # must set
  sonarToken: ""      # must set

keycloak:
  adminUser: admin
  adminPassword: ""

mongodb:
  username: jeeb
  password: ""

nexus:
  adminPassword: ""

sonarqube:
  adminPassword: ""

grafana:
  adminPassword: ""

kong:
  keycloakPublicKey: |
    -----BEGIN PUBLIC KEY-----
    <fill after Keycloak is up>
    -----END PUBLIC KEY-----
```

### `k8s-manager/env/credentials.yaml.example`
Committed template — same structure as above with placeholder comments.

### `k8s-manager/env/.gitignore`
- Replace `credentials.env` entry with `credentials.yaml`

### `k8s-manager/internal/credentials/loader.go`
- Replace `parseEnvFile` with YAML unmarshalling (`gopkg.in/yaml.v3` — already indirect dep via k8s.io)
- Add field: `GrafanaAdminPassword string`
- Keep same `Credentials` struct field names (only the file format changes)
- Keep same required-field validation

### `k8s-manager/internal/setup/secrets_values.go` (NEW FILE)
New file with one exported function:

```go
// GenerateSecretsValues builds a YAML []byte from credentials.
// It also computes nexus.dockerConfigJson (base64 dockerconfigjson)
// from the nexus admin password so that value is never stored at rest.
func GenerateSecretsValues(creds *credentials.Credentials) []byte
```

Generates the YAML structure below and returns it as `[]byte`:
```yaml
mongodb:
  credentials:
    username: <MongoDBUsername>
    password: <MongoDBPassword>
keycloak:
  credentials:
    adminUser: <KeycloakAdminUser>
    adminPassword: <KeycloakAdminPassword>
nexus:
  dockerConfigJson: <base64({"auths":{"localhost:30050":{"auth":"<base64(admin:NexusAdminPassword)>"}}})>
jenkins:
  credentials:
    adminPassword: <JenkinsAdminPassword>
    githubUser: <JenkinsGithubUser>
    githubPat: <JenkinsGithubPAT>
    nexusUser: <JenkinsNexusUser>
    nexusPat: <JenkinsNexusPAT>
    sonarToken: <JenkinsSonarToken>
sonarqube:
  credentials:
    adminPassword: <SonarQubeAdminPassword>
kong:
  keycloakPublicKey: <KongKeycloakPublicKey>
grafana:
  adminPassword: <GrafanaAdminPassword>
```

### `k8s-manager/internal/setup/setup.go`
- In `Run()`, call `GenerateSecretsValues(r.creds)` and write result to `<outputDir>/values-secrets.yaml` (mode 0600) before any deploy step.
- In `deployInfra()`: remove all `--set` flags for credentials, add `-f <outputDir>/values-secrets.yaml`
- In `deployApp()`: remove all `--set` flags for credentials, add `-f <outputDir>/values-secrets.yaml`
- In `deployObs()`: add `-f <outputDir>/values-secrets.yaml`

### `k8s-manager/env/.gitignore`
- Add `values-secrets.yaml`

## Phase 3 — New k8s-manager commands (improvements)

### `k8s-manager validate`
New command that reads `credentials.yaml`, checks every required field is non-empty, and prints a pass/fail table. Run before setup to catch missing values early.
```
[✓] JENKINS_ADMIN_PASSWORD   set
[✓] JENKINS_GITHUB_PAT       set
[✗] KONG_KEYCLOAK_PUBLIC_KEY not set — run 'k8s-manager kong-key' after Keycloak is up
```

### `k8s-manager kong-key`
New command to update the Kong Keycloak public key without re-running full setup:
1. Reads `credentials.yaml`
2. Accepts `--key` flag (PEM string) or prompts to fetch from `http://localhost:30081/realms/jeeb/protocol/openid-connect/certs`
3. Writes updated `credentials.yaml`
4. Re-generates `values-secrets.yaml`
5. Runs `helm upgrade jeeb-infra` for the infra chart only (Kong redeploy)

This replaces the current manual "fill in credentials.env → re-run full setup" flow.

### Auto-apply CoreDNS patch in setup
Currently step 9 prints instructions but does nothing. Improve it to:
1. Detect ingress ClusterIP (already done)
2. Read `k8s/coredns-patch.yaml`, substitute the IP in memory (no file write needed)
3. Pipe the updated YAML to `kubectl apply -f -`
This makes setup fully non-interactive.

## Phase 4 — Template reuse via _helpers.tpl

### `k8s/charts/jeeb-app/templates/_helpers.tpl` (NEW FILE)

Define three named templates:

#### `jeeb-app.vaultAgentHCL`
Renders the shared `agent.hcl` content (vault address, kubernetes auto_auth, file sink, template block). Accepts a dict with keys: `vaultAddr`, `role`, `envFile`.

```
{{- define "jeeb-app.vaultAgentHCL" -}}
vault {
  address = "{{ .vaultAddr }}"
}
auto_auth {
  method "kubernetes" {
    mount_path = "auth/kubernetes"
    config = {
      role = "{{ .role }}"
    }
  }
  sink "file" {
    config = {
      path = "/app/env/.vault-token"
    }
  }
}
template {
  source      = "/vault/templates/env.tpl"
  destination = "/app/env/{{ .envFile }}"
}
{{- end -}}
```

#### `jeeb-app.vaultAgentContainer`
Renders the vault-agent init container spec. Accepts a dict with keys: `vaultImage`, `configMapName`.

#### `jeeb-app.vaultVolumes`
Renders the three vault-related volumes (vault-config, vault-templates, app-env). No parameters needed.

### Refactor vault-agent-config.yaml files (Phase 4)
Replace the identical HCL boilerplate in all three with `include "jeeb-app.vaultAgentHCL"`:

**`backend/vault-agent-config.yaml`:**
```yaml
data:
  agent.hcl: |
    {{- include "jeeb-app.vaultAgentHCL" (dict
        "vaultAddr" .Values.infra.vaultAddr
        "role"      .Values.backend.vault.role
        "envFile"   .Values.backend.vault.envFile) | nindent 4 }}
  env.tpl: |       ← backend-specific env vars only
```

Same pattern for frontend and learning — only the `env.tpl` section differs per service.

### Refactor deployment.yaml files
Replace the duplicated vault-agent init container and volumes in `backend/deployment.yaml`, `frontend/deployment.yaml`, `learning/deployment.yaml` with `include` calls to the new named templates.
